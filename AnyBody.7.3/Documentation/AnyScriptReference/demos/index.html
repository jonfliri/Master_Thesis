Demo examples¶

This chapter contains a selection of small example models which demonstrates different parts of the AnyScript language. The models are divided into three sections:

Small Demo Models:

These are simple examples of very basic biomechanical and mechanical applications to demonstrate basic concepts of modeling

Class Examples:

These are examples demonstrating the use and basic behaviour of AnyScript™ classes

Basic AnyScript™:

These examples demonstrate basic AnyScript™ features such as the preprocessor, mathematical expressions in AnyScript™, the macro language and alike.

SliderCrank2D Planar slider crank mechanism.¶

Description

Planar slider crank mechanism.

Download

Demo.SliderCrank2D.zip

Demo: SliderCrank2D

This file contains a model of a planar slider-crank linkage Notice that the mechanism is kinematically over-determined according to the joint configuration. Motion is however possible because some of the kinematic constraints are not independent, i.e. the system contains redundant constraints.

The model demonstrates the definition of revolute (hinge), spherical (ball-and-socket), and prismatic (translational) joints.

Notice how the Coupler ‘s local reference axes are oriented quite differently than the other segments. The Coupler is the red segment. It has the y-axis, not the x-axis, being normal to the mechanism’s plane. This is realized by use of relative orientation of a node on the segment (AnyRefNode). This feature is needed in systems where the axes of joints are not coinciding well on the connected segments; in joints rotation/translation axes are typically specified by simple a x, y, or z specification defining the axis on both segments. Thus proper alignment must be obtained rotation of the nodes (e.g. by means of AnyRefNode) the joint connects.

Moreover, the models applies a simple kinematical driver by using an AnyKinDriver object.

Drivers Demonstrating of different kinematical driver types.¶

Description

Demonstrating of different kinematical driver types.

Download

Demo.Drivers.zip

This file demonstrates a number of specialized kinematic drivers derived from the class AnyKinEq. They are all applied to the rotation a single segment.

Please notice that a new generic driver class, AnyKinDriver, is available. This AnyKinDriver can replace most of the older specialized driver classes, because it allows pointing to a driver function. It is recommended to use this new class.

The following drivers exist:

  1. AnyKinEqPolynomialDriver

    It provides a polynomial function of any order, depending on the number of coefficients you supply. The function has the form: p = sum_{i=0}^{n} PolyCoef[i] t^i (LaTeX style)

  2. AnyKinEqFourierDriver

    It provides Fourier functions of any order. You can chooce between expansions containing

    1. A cos(omega t) + B sin(omega t)

    2. A cos(omega t + B)

    3. A sin(omega t + B)

    where omega = 2 PI Freq with Freq as the fundamental frequency of the expansion.

  3. AnyKinEqInterPolDriver

    This driver interpolated between data points assembled in the matrix Data. Corresponding time values must be supplied in the vector T. The interpolation driver have several options that are demonstrated several objects in the model below. These options are

    • Piecewise linear interpolation between the data points (notice that accelerations are zero implying that this is only good for slow motion with insignificant dynamic effets).

    • Bezier interpolation

    • B-spline interpolation with variable order (the lowest valid order is 4 in order to ensure continuity in the accelerations).

    • Data points provided in external data file

  4. AnyKinEgSimpleDriver

    This provides a simple driver function with constant acceleration, i.e., constant second time derivatives.

Notice that this script declares at least one of each type of driver. They are all located in the Main.Drivers folder. Associated with each driver there is declared a study (AnyMechStudy) which contain the mechanical elements, the joint and segment, and the particular driver. You can run an analysis using a given driver function by excuting the kinematic analysis of the associated study.

Notice that these drivers are derived from AnyKinEq. However, in the future drivers will be handled by a single driver class and a library of general functions and mathematical expressions, which can be used to specifying the motion.

AnyDesEvalStudy Demostration of the use of design evaluation studies together with optimization studies.¶

Description

Demostration of the use of design evaluation studies together with optimization studies.

Download

ClassExamples.AnyDesEvalStudy.zip

In this model we use the AnyDesEvalStudy to set the initial driver angle for the crank in the Demo.SliderCrank2D.any example. We want to specify this initial angle so that the slider is in a given position.

We add a new mechanical study with a new kinematic analaysis. In this we remove the crank driver and insert a driver for the slider which can position the linkage in the wanted position. The design-evaluation study uses this new kinematic analysis to find the new angle and then it tranfers the angle value from crank joint Pos member to the crank driver’s DriverPos member.

Subsequently, the kinematic analysis of the original MechStudy from Demo.SliderCrank2D.any will start from not zero angle, but the newly found angle.

Please try to

  1. load the model

  2. Run Main.MechStudy.Kinematics

  3. Run Main.RunAll

… and watch the difference.

AnyForce AnyForce¶

Description

AnyForce

Download

ClassExamples.AnyForce.zip

Demo: Force and Force Measures

This file demonstrates the application of forces to a mechanical system and how these loads can practically be monitored.

The model consist of a simple, two-link manipulator with revolute joints connecting the links and the base unit (ground).

We apply the following loads to the end-effector:

  • A global force, i.e. a vector of force in global coordinates.

  • A local force, i.e. a vector in given in the reference frame of the end-effector.

  • A moment/torque, given as a vector

Moreover, we eliminated gravity and intertia forces by specifying very small masses. All quantities have be selected simply to illustrate; they are not related to any practical application.

We will measure the effect of the applied loads at two points of interest, the end effector and the base joint. We do this by the class AnyForceMomentMeasure, which provides us with a total equivalent load given as a single force and moment vector in global coordinates.

To validate the results, we compute the effect of the loads manually, i.e., by means of the mathematical expressions. These expressions can be found in the InverseDynamicAnalysis folder in the LoadStudy. Here they will evaluated during the analysis, and we can find the results in the Chart View.

This also demonstrates the possibility generating your own derived output by expressions.

We have placed the AnyForceMomentMeasures directly in the LoadStudy, so that you can easily compare their results with the one in InverseDynamicAnalysis.

Use for instance an asteriks character in the Chart View specification like “LoadStudy.Output.*.M” to compare the components of the moments. Or use more than one asteriks, like “LoadStudy.Output.*A*.M” to access object with names containing an ‘A’, i.e., InverseDynamicAnalysis and BaseJointAndDriver.

AnyForceConstraint Applying constraints on forces, in particular reaction force.¶

Description

Applying constraints on forces, in particular reaction force.

Download

ClassExamples.AnyForceConstraint.zip

An example showing different force constraints (AnyForceConstraint-derived classes)

The example is based on the Demo.Arm2D model, however modified with new drivers. The new drivers and force constraints aim at modeling a vertical wall with a sliding box on. The box is held by the hand of the simple 2D arm. Between box and wall exists a Coulomb-style friction. Although, we have motion between box and wall we make the friction model as in-equality constraint between normal and tangential forces (in fact a cone constraint), i.e., static Coulomb friction. We take this freedom for demonstraing purposes.

Basically, it constains the following studies:

ArmModelStudy : The original study imported with the original model and not relevant for this example

ArmModelStudy2 : The arm model assembled with the wall-box-friction model. The original drivers excluded

ArmModelStudy2_NoFriction : The same as ArmModelStudy2; however, the box and friction is removed.

The RunApplication operation runs the inverse dynamics operation of the two studies. Comparing the studies we see how the arm exploits the friction for carrying the load on the arm, but only until a certain height (moving the hand down). Lower than this height, the two models show the same muscular effort (apart from the weight of the box).

Download

ClassExamples.AnyKinMeasureOrgAndAnyKinEq.zip

Demo: Organizing kinematic measure and constraints

In this demonstration model, we shal illustrate how you can kinematically drive a single coordinate of motion. The idea is to reorganize kinematic measures, in this case picking out the quantities you want to constrain. We do this by making a new measure that is a reorganization of existing ones. Subsequently, we can drive this new measure, i.e., only the selected coordinates.

The model is simple: It is a single segment attached to ground by a hinge (AnyRevoluteJoint). It is extracted from another file with a model of the arm, thus the segment is called UpperArm. The dimensions in the problems indicate that we are dealing with a giant, but please do not think too much about this.

The motion is generated by moving the Y-coordinates of the elbow, i.e, we drive a single coordinate of the elbow motion.

AnyShortestPathMuscle Muscle wrapping by class AnyShortestPathMuscle.¶

Description

Muscle wrapping by class AnyShortestPathMuscle.

Download

ClassExamples.AnyShortestPathMuscle.zip

Demo: AnyShortestPathMuscle.any

The model demonstrates the special class for making muscles that needs to wrap over obstacles. This class, AnyShortestPathMuscle, uses the AnyKinSPLine (SP=shortest path) as internal kinematic measure. Here we demonstartes how to define the shortest-path muscle, enter origin, insertion and via-points, and obstacles for wrapping about. A final and very important issue is initial positioning of the string. The initial positions may determine which way about obstacles the final path will be, in case of multiple valid configuration.

Abut initial positioning notice the following: 1) InitWrapPosVecArr is new style of defining initial position for the wrapping, which is fra more flexible than the old style using InitWrapPosVectors.

2) Setting InitWrapPosOnlyOnOff=On will only calculate initial position but omit the real wrapping. This way the initial positions can be inspected during the motion of the model.

  1. InitWrapPosVectors is the old style of definition. It will become deprecated.

Using the #define MUSCLE1_VIA_POINT, the example includes a via point for the muscle
