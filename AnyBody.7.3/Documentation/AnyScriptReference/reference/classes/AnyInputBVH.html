AnyInputBVH¶

Type

Creatable

Parent class

AnyDataFileBase

Description¶

Class to read files in BVH format

The class reads data in a BVH file, and optionally, it can create a model with segments resembling the kinematical hierarchy in the file. Data from long recordings can be cropped during the import by members StartFrame and EndFrame.

During import it can apply scaling to all linear and rotational quantities independently. Two scaling parameters, TranslationScale and RotationScale, exist for scaling of translations and rotations, respectively. This is useful for files using other units than meters and radians, which is basic unit set in most models. The scaling is applied to the data while reading it into the AnyScript data structure, so all data you find in the AnyInputBVH objects are scaled.

In additional to data scaling, it is also possible to define filters for different types of motion data in the file (e.g. translation, rotational). By default, one filter (called AnyInputBVH.Filter) is included in the AnyInputBVH object, but additional filters can be defined simply by constructing them inside the object or by creating references or pointers (more details below).

Special filter-options for angular data (Rotation2PiFixOnOff, Rotation1PiFixOnOff, and RotationFixByModelOnOff) allow to remove discontinuities arising from angular values jumping any amount of full rotations or discontinuities related multiple configurations for the Euler angle definition. This is particularly problematic for BVH files with angle definitions that imply that actual orientations are close to Gimball Lock. It is always recommended to avoid Gimball Lock configuration, but using standard system’s BVH files this can be impossible. Please find more details on this below.

The AnyInputBVH object can construct a mechanical model of the the kinematical chains of the BVH hierarchy. The ConstructModelOnOff switch enables this and the constrcuted model is placed in a subfolder folder called Model. This is on by default. Construction of the model is also associated with several options for specifying model visualization of model, the socalled BVH stick figure. It is also possible to set which time frame (InitialPos_FrameNo) it should use to generate the model’s initial configuration. This is if only a subsection of a long recording is use to start the model close to your actuall start point.

New in version 7.2.3: In AnyBody v.7.2.3, a number of new options where released (pre-release for AnyBody, v.7.3). This was released so that AnyInputBVH currently can run in two modes: A compatibility mode for version 7.2 and a new, recommended mode. The new mode contain better options to deal with jumping angular data and virtual marker based interface to a human model. A switch, CompatibilityMode_72, determines the mode and it value will also control warnings and Model Tree visibility so that parameters associated with the inactive mode are hidden and warnings will be given in case of usage.

The new mode (version 7.3) is characterized by the following:

  • Virtual marker data can be generated by the AnyInputBVH object. It uses (requires) the in-object constructed model (Model folder) which is simulated in a hidden process inside AnyInputBVH. The virtual marker data is collected in the folder called Marker (optional). Marker data spaces is allocated by specifying the name of your virtual marker in MarkerNames array. Each marker must be linked to an actual reference frame (AnyRefFrame), typically a node on one of the segments in the BVH model (Model folder). Such nodes can be created as AnyRefNode objects inside the segments of the model and subsequently linked to the marker by RefFramePtr member in the marker, i.e. Marker.<MARKER_NAME>.RefFramePtr.

  • Filtering of data can be applied to translational and rotational channel data, as well as the data for the virtual markers. Three object-pointers (MarkerFilter, TranslationalDataFilter, RotationalDataFilter) enable you to select different filters for each of these (Previous filter specification, see below, is deprecated). It is recommended to use a marker-based interface to the human model and to apply filtering to marker data. Therefore, the filters for channel are set to Node by default as implied by this recommendation.

  • Rotation fix (discontinuity) filtering has a new option, i.e. to “fix” angle jumps by using the BVH Model. This option, RotationFixByModelOnOff, use angular kinematic measures in the BVH Model (Model folder) to measure angular data and thereby to improve consistency of the data with these kinematic measures when they are later used to drive the model. This makes the option Rotation2PiFixOnOff and Rotation1PiFixOnOff obsolete (Off by default in this mode). The output of the process is collected in ProcessedMotion members in the Data folder.

    Please notice, that none of the angle fix options can guarantees removing angles jumps. Occasioanlly, BVH files contains jumps in Euler angle configuration stored over several samples. AnyInputBVH may be recognize this as a continous angle variation and therefore not apply any modifications. This makes the data inappropiate for filtering and potential even for kinematic analysis of the model, especially close to Gimball Lock configurations. These problems are removed by using a marker based interface to the human model, which will also eliminate the need for kinematic analysis of the BVH model (only in this new mode).

  • In this mode, BVH model drivers has been changed to standard AnyKinDriver connected with data interpolation function in the Data folder. This replaces dedicated, old-style interpolation drivers (AnyKinEqInterPolDriver), which also doubles the data storage.

Compatibility mode (AnyBody v7.2) is characterized by the following:

  • Virtual marker data, and filtering thereof, is not available inside the AnyInputBVH object. The data can be generated by means of a kinematic analysis of the BVH model (Model folder), which however is failure prone with poor angular data as explained above.

  • Specification of which filter to use is made using translationalDataFilterIndex and RotationalDataFilterIndex for the translational and rotational data, respectively.

  • Rotation fix (discontinuity) filtering options (Rotation2PiFixOnOff, Rotation1PiFixOnOff) are On per default. This angular discontinuity filters will first remove discontinuities of 2Pi (if switched on) and subsequently, discontinuties of Pi related switching Euler angles configurations (if switched on). Unfortunately, these filter are not able to detect all configuration switches correctly. For instance, many BVH files contain such discontiniuities sampled over several time frames making them impossible to detect. In case of problems with your data, it is recommended to switch to the new mode and use marker based interface to the human model.
