AnyKinSPSheet¶

Type

Creatable

Parent class

AnyKinPLine

Description¶

This “kinematic measure” can be used to model a sheet spanned by reference frames and measures the length of shortest path (SP) lines in the sheet. The sheet may be in contact with surfaces (AnySurface), contain linear inequality constraints to restrict positions of wrapping points and may contain distance constraints between wrapping points.

The sheet is defined as a number of polylines (see AnyKinPLine) in longitudinal direction. Each segment of a polyline contains a number of wrapping points (see AnyKinSPLine) defined in MeshLong, where the number of wrapping points can be defined for each segment line segment individually or for equal all line segments. The number of wrapping points in the lines of the sheet are all equal. In lateral direction these lines are connected by linear springs in each wrapping point.

The stiffness of elements is defined by StiffnessLong and StiffnessLat for the longitudinal and the lateral direction respectively. These values define the overall stiffness of a segment of the sheet in longitudinal direction and lateral direction. The stiffness’s of the elements in the sheet with \(m\) longitudinal wrapping points in a sheet segment and \(n\) lines are thus

\[k_{long,el} = k_{long} \frac{m + 1}{n}\]

and

\[k_{lat,el} = k_{lat} \frac{n}{m+2}.\]

The element stiffness can be changed individually by setting StiffnessRatiosLong and StiffnessRatiosLat. These matrices with entries \(a_{i,j}\) define ratios used to updated the element stiffness by

\[k_{long,el i,j} = k_{long,el} \cdot a_{i,j}\]

for entries in StiffnessRatiosLong and

\[k_{lat,el i,j} = k_{lat,el} + a_{i,j} k_{long}\]

for entries \(a_{i,j}\) in StiffnessRatiosLat. The sizes of the matrices have to fit the number of elements in the complete sheet, i.e. for StiffnessRatiosLong

\[\left(\sum_{seg} (m_{seg} + 1)\right) \times n\]

for and StiffnessRatiosLat and

\[\left(\sum_{seg} (m_{seg})\right) \times (n-1)\]

with the number of wrapping points in longitudinal direction on a line \(m_{seg}\) as defined in MeshLong. By default, all entries of StiffnessRatiosLong are set to 1 and the entries of StiffnessRatiosLat are set to 0.

The longitudinal lines of the sheet can be in contact with and wrap over rigid surfaces (AnySurface). This wrapping can be switched on or off for the lines in the sheet by setting the matrix SurfaceSwitches. If defined, this matrix has to contain a vector of 1 and 0 for each line in the sheet indicating if a line is wrapping on a surface or not. For each wrapping surface the matrix one line has to be defined. By default, all lines are wrapping on all surfaces.b The contact between sheet and surfaces is considered as frictionless and is governed by static equilibrium, i.e., minimum potential elastic energy of the sheet.

The shape of the sheet can be controlled by distance constraints between wrapping points in longitudinal and/or lateral direction defined in ConDistLongIndex, ConDistLongPointSwitch and ConDistLongValue for constraints in longitudinal direction and ConDistLatIndex, ConDistLatPointSwitch and ConDistLatValue for constraints in lateral direction. A constrained is defined by an index of the column (i.e. index of line in the sheet) or row (i.e. index of wrapping point on a longitudinal line) for longitudinal or lateral distance constraints, resp., an index vector with zeros and ones indicating the use of a point in the constraint and value \(b\) of the constraint by

\[\sum_i d_i^T d_i - b \geq 0\]

with \(d_i = x_{\phi(i)} - x_{\phi(i+1)}\) where \(x_k\) is defined as the position of the \(k\)-th point in corresponding row or column and \(\phi(i)\) is the index of the \(i\)-th nonzero entrie in the PointSwitch. The length of rows in the PointSwitch matrices has to be equal to the number of wrapping points in longitudinal or lateral direction of the complete sheet, resp.

The shape of the sheet can also be controlled by linear constraints on the position of individual wrapping points. For a point \(x\) defined in ConLinPointIndex, the position of a reference frame \(p\) in ConLinRefFrame and a direction \(N\) defined by an AnyVec3 \(n\) defined in ConLinDir defined in a reference frame given in ConLinDirRefFrame, the linear constraint is defined as

\[N^t (x-p) \leq 0.\]

For the specification of the PointIndex, the wrapping points in the complete sheet are numbered row-wise starting with index 0. The reference frames spanning the sheet are not part of this numbering.

The contact problem is solved iteratively until the displacement of the summed stringlengths in the sheet between iteration steps is less than a given tolerance. For solving the contact problem, two different solvers are available which can be specified in SolverType:

  • A solver SPSolverFull handling the problem calculating a solution of all points in the sheet. This method is a more robust option.

  • A solver SPSolverModes solving a reduced problem containing only constrained points and calculating the position of all other points from the position of the constrained points. This solver is used as default option. In general, this solver is fast but can in some cases be less robust. Thus, the SPSolverFull is used as fallback. The use of the fallback solver will be indicated by a warning. Due to this fallback, for some models this solver can be slower than using the SPSolverFull.

In some cases, the performance of the solver can be increased by changing the parameter ContactActivationDistance which defines an activation distance between wrapping points and surfaces for wrapping constraints. This parameter should be handled as an “expert parameter” and it is recommended to use it unchanged for most models.

The position for calculating the initial conditions of the sheet are similar to the definition of the inital position of an AnyKinSPLine and can be defined using the reference pointer array InitWrapPosVecArr. If used, this array has to contain as many pointers to 3D AnyFloat arrays as segments are defined in the sheet. Each array can contain an arbitray number of matrices definining an initial position point for each line in the sheet segment. Thus, each matrix needs to have 3 columns and \(m\) rows. The 3D vectors define points in the global reference frame. For debugging a model, the switch InitWrapPosOnlyOnOff can be set to On which switches of the wrapping and constraint handling off and calculates the sheet only going through the inital positions. With the switch InitWrapPosForEachStepOnOff, the algorithm can be evoked to use the initial conditions in as initial guess in every time-step.

The result of the measure are positions and velocities of the length of all longitudinal lines in the sheet.

Known issues

This is a very challenging functionality to implement and the numerical and iterative nature of the evaluation of this kinematic measure means that it is not a 100 percent stable. Moreover, it demands relatively much computer power compared to typical “kinematic measures” that are based on simple mathematical expressions that are straight forward to evaluate. AnyKinSPSheet objects can slow down models significantly.

Large time step, i.e., steps where large motion has occured, can cause problems. The numerical solution relies on initial guesses for the position of the string. These are entered manually for the starting point in time, but during an time analysis, the previous time step and information about the motion is used to create the guess of the position for the next step. Large time steps can therefore cause these guesses to be inadequate, which can make the position analysis fail to converge or converge to an unexpected solution causing jumps of the string position.

The velocity analysis may fail with an exception reporting “negative pivot” during a factorization. This can happen when the wrapping geometry is not defined properly and segments of the polyline penetrates a surface too much. The direct cause of this can be that an endpoint or via-point of the string penetrates the surface or that the mesh of the string is too coarse. In the former case a warning about the penetration will be given, however, the penetration is accepted by the system unless the “negative pivot” occurs.

Due to the complexity of the underlying mathematical problem, the acceleration analysis for this measure is not implemented. This implies that this measure cannot be used for kinematic drivers, however it is sufficiently equipped to function for definition of forces and thereby muscles, which is the primary objective of the kinematic measure.

Future versions

This class and the underlying numerical solver routines are still undergoing development. Increased robustness, easier user control, and increased efficiency are areas of the continous research and development in this area.

This implies that changes of this class must be expected in future versions of AnyBody.
